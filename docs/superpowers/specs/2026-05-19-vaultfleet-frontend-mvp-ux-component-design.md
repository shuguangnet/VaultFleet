## 1. 整体架构与导航

VaultFleet 采用典型的 **Sidebar Layout**（侧边栏布局），旨在提供一种“工具感”强、操作高效的运维体验。

### 1.1 导航结构 (Sidebar)
- **Dashboard (仪表盘)**: `/`
- **Nodes (节点管理)**: `/nodes`
- **Storage (存储配置)**: `/storage`
- **Policies (备份策略)**: `/policies`
- **Tasks (任务历史)**: `/tasks`
- **Snapshots (快照浏览)**: `/snapshots`
- **Notifications (通知设置)**: `/notifications`
- **System (系统管理)**: `/system`

### 1.2 顶部栏 (TopBar)
- **面包屑导航**: 展示当前所在页面路径。
- **系统状态摘要**: 简报（如：3 Online / 5 Total Nodes）。
- **用户菜单**: 仅提供“修改密码”入口（注：当前无 logout 接口，不展示退出登录入口，避免呈现半成品状态）。

## 2. 页面级 UX 设计

### 2.1 初始化与登录 (Auth Flow)
- **Setup 页面**: 仅包含 `用户名`、`密码`、`确认密码`。大标题“欢迎使用 VaultFleet”。（注：确认密码仅在前端进行本地校验，后端 `POST /api/auth/init` 仅接收 `username` 和 `password`）。
- **Login 页面**: 极简风格，错误信息实时反馈。

### 2.2 仪表盘 (Dashboard)
- **状态卡片 (Metric Cards)**:
    - 节点状态：在线/离线比例。
    - 策略状态：已同步/未同步数量。
    - 任务统计：24小时内成功/失败计数。（注：后端暂无时间筛选接口，前端需通过 `GET /api/tasks?limit=200` 拉取数据后，根据 `created_at` 或 `finished_at` 在本地进行 24h 范围过滤与统计）。
- **最近任务 (Recent Tasks)**: 展示最新的 10 条备份或恢复记录。点击行可跳转至 Tasks 页面并自动应用对应 Agent 或状态的筛选视图（注：由于后端无单任务详情接口，不设计独立详情页）。
- **快速入口**: 提供“添加节点”和“创建策略”的显著按钮。

### 2.3 节点管理 (Nodes)
- **列表页**:
    - 支持按名称搜索。
    - 状态列使用 `StatusBadge`（Green/Red）。
    - 操作列：详情、重新生成安装 Token、删除。
    - **安全机制**: “重新生成安装 Token”操作必须触发确认弹窗，并明确提示该操作会使旧 Token 失效。
- **添加节点 (Add Node)**:
    - 统一采用 **Drawer**（侧滑抽屉）。
    - 填写名称后，点击“生成”，界面转换为安装指令面板。
    - **InstallCommand 组件**: 展示完整脚本，支持一键复制。
- **节点详情 (Node Detail)**:
    - 采用 Tab 切换：
        - `概览`: 基础系统信息。
        - `策略`: 关联的备份策略列表。
        - `快照`: 该节点的 restic 快照。
        - `任务`: 该节点的历史任务。
        - `文件浏览器`: **核心功能**，支持实时浏览 Agent 上的目录，点击可快速复制路径到策略配置。

### 2.4 存储配置 (Storage)
- **列表页**: 展示存储类型（S3, WebDAV 等）和名称。
- **表单设计**:
    - 统一采用 **Drawer**（侧滑抽屉）进行创建和编辑。
    - **双模式编辑器**：
        - **模版模式**：针对 S3, WebDAV, SFTP, Local 提供常用字段的友好输入界面。
        - **高级模式 (Key-Value Editor)**：提供通用的键值对编辑器，确保支持任何自定义 rclone 存储类型及高级参数配置。
    - 敏感字段（Password/Secret）默认掩码处理。
    - **极简操作**：仅展示“保存”操作，不渲染任何“测试连接”按钮。

### 2.5 备份策略 (Policies)
- **列表页**: 重点展示 `Sync 状态`。如果 Master 修改了配置但 Agent 尚未拉取，显示 Amber 色的“待同步”。
- **策略编辑器**:
    - 统一采用 **Drawer**（侧滑抽屉）进行创建和编辑。
    - 包含：Agent 选择、存储选择、备份路径（多行输入或通过文件浏览器选取）、排除规则、Cron 调度。
    - **Retention (保留策略)**: 提供简单的输入项（Keep Daily/Weekly/Monthly）。

### 2.6 任务历史 (Tasks)
- **高级过滤**: 支持按 Agent、类型（备份/恢复）、状态（成功/失败）筛选。
- **详情展示**: 展开行展示完整的 `Error Log`。对于成功任务，展示 Repo Size 变化。

### 2.7 快照浏览 (Snapshots)
- **页面结构**: 顶部提供 Agent 选择器（仅展示所选 Agent 的快照，不做跨节点聚合）。
- **快照列表**: 展示 快照 ID、时间、包含的路径、大小。
- **操作**:
    - **刷新快照**: Agent 在线时可用，展示 Loading 状态和超时处理。
    - **恢复**: 单行点击唤起 Restore 流程。

### 2.8 恢复流程 (Restore)
- **触发入口**: 从 Snapshots 列表或节点详情的单行操作触发。
- **表单设计 (Drawer)**:
    - 只读展示：被选中的 Snapshot ID。
    - 输入项：目标路径 (Target Path)。
    - **安全提示**: 明确提示恢复动作将在 Agent 所在机器上执行，必须确认目录可写。
    - **确认机制**: 必须勾选“确认恢复”复选框后方可提交。
- **提交后**: 展示已接受状态及 Message ID，并引导跳转至 Tasks 页面查看进度。

### 2.9 通知设置 (Notifications)
- **列表页**: 展示已配置的告警规则。
- **编辑器**:
    - 统一采用 **Drawer**（侧滑抽屉）进行创建和编辑。
    - 类型选择：Telegram 或 Webhook。
    - 事件选择：支持多选，第一版仅提供 `backup_failed` 和 `agent_offline`。
    - 动态表单：依据类型展示特定字段（如 Telegram 的 Bot Token / Chat ID）。
    - **测试通知**: 提供显著的“发送测试”按钮以验证配置连通性。

### 2.10 系统管理 (System)
- **页面结构**: 基础区块罗列。
- **修改密码**: 表单包含当前密码、新密码、确认密码。
- **数据导出**: 提供显著的“导出 Master 数据”按钮，点击触发 `.zip` 数据库备份下载。
- （注：当前无版本或审计日志接口，因此不渲染相关占位内容）

## 4. API 集成与响应归一化

由于后端接口存在不一致性，前端将建立统一的 `ApiClient` 层：

1. **响应转换与错误归一化**:
    - 以下情况均需转换并抛出前端错误：
        - HTTP 响应状态码非 2xx。
        - 响应体包含 `{ ok: false, error: "..." }`。
        - 响应体直接包含 `{ error: "..." }`。
    - 自动解包 `{ data: [...] }` 并适配 `notifications` 等直接返回数组的接口。
2. **认证管理**:
    - 每次请求携带 `credentials: "same-origin"`。
    - 401 状态自动重定向至 `/login`。
3. **加载状态管理**:
    - 严格依赖 `TanStack Query` 实现自动重试、缓存和全局 Loading 状态追踪。

## 5. UI 文案规范 (中文)

- **动作**: `添加`、`编辑`、`删除`、`立即备份`、`刷新快照`、`恢复数据`。
- **状态**: `运行中`、`已完成`、`失败`、`在线`、`离线`、`待同步`。
- **提示**:
    - “正在请求 Agent 浏览目录，请稍候...”
    - “注意：恢复操作将覆盖目标路径下的同名文件。”
    - “此密码仅在创建时显示一次，请务必妥善保存。”

## 6. MVP 验收视觉标准

- 页面切换无明显闪烁（强制使用 `TanStack Query` 管理加载状态与缓存，避免路由层阻塞或多套状态方案并行）。
- 所有破坏性操作（Delete）必须有 ConfirmDialog。
- 所有表单必须有基础的客户端校验（如密码长度、必填项）。
- 手机端能正常查看状态和列表（Responsive Design）。
- 错误信息能够清晰、准确地展示后端返回的 `error` 文本。
