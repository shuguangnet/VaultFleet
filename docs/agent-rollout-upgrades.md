# Agent 批量升级与灰度发布

VaultFleet 支持在 **节点管理** 页面创建 Agent 批量升级任务。该能力用于多节点环境的受控升级：先升级少量 canary 节点，确认 Agent 重新连接并上报目标版本后，再按批次升级其余节点。

## 适用场景

- 多台 VPS、OpenStack 或多地域节点需要统一升级 Agent。
- 希望先验证 1 台或少量节点，避免坏版本一次性扩散。
- 需要按节点标签选择目标，例如 `prod`、`web`、`region:sg`。
- 需要保留升级过程、失败原因和跳过原因，便于排障和审计。

## 创建升级任务

在 **节点管理** 点击 **批量升级**，配置：

- 目标版本：例如 `v0.5.42`。留空时使用系统默认版本或 latest 行为。
- GitHub 仓库：留空时使用系统默认仓库。
- 目标标签：选择带有指定标签的节点。
- 目标节点：可显式选择单个或多个节点；会与标签匹配结果合并去重。
- Canary 数量：先升级的节点数量，默认 1。
- 批大小：canary 成功后，每批并行升级的节点数量，默认 5。

创建前页面会展示目标预览，包括节点、在线状态、当前版本和架构。只读用户不可创建或取消升级任务；管理员和运维角色按节点写权限控制。

## 执行语义

升级任务由 Master 侧协调器持久化执行：

1. 解析目标标签和显式节点，生成每个节点的升级项。
2. 离线、未知版本或未知架构的节点会在预检阶段标记为跳过或阻塞原因，不会无限等待。
3. 已经处于目标版本的节点会直接标记成功，不再发送升级命令。
4. Master 先向 canary 节点发送现有 `update_agent` 消息。
5. Agent 接受升级请求后，升级项进入运行状态。
6. 只有当 Agent 重启或重新心跳并上报目标版本时，该升级项才算成功。
7. 所有 canary 成功后，才会按批大小启动后续批次。
8. 当前批次全部成功后，才会启动下一批。

`update_agent` 的 ACK 只代表 Agent 接受了升级请求，不代表下载、替换、重启和重新连接已经完成。因此 VaultFleet 以目标版本心跳作为成功依据。

## 失败停止规则

任一运行中的升级项发生以下情况时，整个升级任务会停止：

- Agent 拒绝 `update_agent` 请求。
- Agent 接受请求后超时未上报目标版本。
- 升级协调器记录到节点级失败。
- 操作者手动取消任务。

任务停止后，尚未开始的节点会标记为跳过，并记录跳过原因。UI 的 **Agent 升级任务** 表会显示任务状态、成功/运行/等待/失败/跳过计数和失败原因，便于判断是否需要修复后重新创建升级任务。

## 与自动更新的关系

VaultFleet 原有心跳链路会根据 Agent 版本信息触发自动版本提示或更新行为。受控批量升级期间，如果某个 Agent 已经属于非终态升级项，Master 会抑制该 Agent 的自动 `version_info` 更新路径，避免未进入 canary 或当前批次的节点绕过灰度控制。

没有活跃升级项的 Agent 仍保持原有自动更新行为。

## 运维建议

- Canary 选择优先用低风险但能代表真实环境的节点，例如同系统、同架构、同部署方式的非关键节点。
- 生产环境建议 canary 数量从 1 开始；确认版本稳定后再扩大。
- 批大小按故障承受能力设置。小型集群可用 2 到 5，大型集群按可接受同时重启数量拆批。
- 混合 `amd64` / `arm64` 环境升级前，确认 Release 中存在对应架构的 Agent 资产。
- 升级失败后先查看失败节点的 Agent 日志、Master 任务状态和网络连通性，再创建新的升级任务。
- 如果节点长时间离线，不要把它作为关键 canary；离线节点会被跳过或阻塞，不会作为成功信号推进批次。
- 需要回滚时，当前版本不自动执行二进制回滚。应指定旧版本重新创建升级任务，或在节点上手动安装旧 Agent。

## 排障

- 显示拒绝升级：检查 Agent 是否支持自更新、安装路径权限和 GitHub Release 下载地址。
- 超时未确认版本：检查 Agent 服务是否重启成功、是否能连接 Master、目标版本字符串是否与 Release 版本一致。
- 后续节点被跳过：这是失败停止策略的预期行为，先修复失败节点，再重新创建任务。
- 页面看不到批量升级按钮：确认当前用户是管理员或运维，并具备节点写权限。
