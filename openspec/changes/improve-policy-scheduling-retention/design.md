## Context

策略表单当前将 `schedule` 作为原始字符串编辑，并将 restic 的 `keep_last`、`keep_daily`、`keep_weekly`、`keep_monthly` 直接展示给用户。Agent 使用 robfig/cron 执行调度，五段式 Cron 会在 Agent 端补秒，实际时区是 Agent 的本地时区。数据库和协议已经稳定保存 Cron 字符串及 retention JSON，因此本次应改善配置与校验，而非引入新的持久化模型。

## Goals / Non-Goals

**Goals:**

- 让普通用户通过业务化控件配置常见备份时间，无需理解 Cron 语法。
- 允许高级用户继续输入已有 Cron，并在编辑时保证未知表达式不被改写。
- 清晰解释多层保留规则是并集关系，而不是备份执行次数。
- 在前后端一致校验调度和保留数量，阻止明显无效策略。
- 保持现有 API、数据库和 Agent 协议兼容。

**Non-Goals:**

- 不允许用户通过“每周 N 次”让系统自动挑选不确定的日期；用户必须明确选择星期几。
- 不在本次变更中增加每个策略的独立时区字段。
- 不改变 restic retention 的分组和清理算法。
- 不实现秒级可视化调度；高级表达式仍可使用 Agent 已支持的格式。

## Decisions

### 1. 可视化模式只生成确定的标准 Cron

调度模式包含“每天”“每周”“每月”“固定间隔”“自定义”。每天选择一个时间；每周选择一至七个星期和一个时间；每月选择日期与时间；固定间隔支持分钟或小时。构建器输出标准五段式 Cron，表单最终仍只提交 `schedule` 字符串。

“每周执行三次”不能唯一确定执行日，因此 UI 将其表达为选择三个星期，而不是一个次数输入。相比自动均匀分配日期，这一方案可预测、可审计，且重新编辑时不会产生隐含规则。

### 2. Cron 与可视化配置之间使用无损识别

前端提供纯函数完成可视化配置到 Cron 的生成、Cron 到可视化配置的识别、描述与预览。只有完全匹配支持模式的表达式才进入可视化模式；其他合法表达式进入自定义模式并原样保存。用户从自定义切换到可视化模式时需要明确选择新规则，不能静默覆盖表达式。

### 3. 校验以 Agent 调度器语义为准

Master 提取共享的 Cron 校验规则，接受现有 Agent 支持的五段、六段及描述符格式，创建和更新策略时均校验。前端执行即时校验以提供快速反馈，但 Master 是最终权威。可视化构建器只生成五段式表达式。

### 4. 时间说明明确采用 Agent 本地时区

由于当前协议未上报可靠的 IANA 时区，UI 将执行时间标记为“节点本地时间”。下一次执行预览展示规则和相对日期时必须同时显示该时区提示，不能暗示它使用浏览器或 Master 时区。

### 5. 保留规则保持四个现有字段并重写语义

保留区域分为预设和自定义两种方式，自定义始终直接编辑四个非负整数：最近 N 次、每日 N 份、每周 N 份、每月 N 份。界面说明这些规则取并集，同一个快照可能同时满足多条规则且只计一份；`0` 表示关闭对应层级。至少一个值必须大于零。

相比只提供“保留最近 N 次”，保留四层规则能维持长期时间点覆盖；相比新增抽象字段，继续提交现有 JSON 可避免迁移与兼容风险。

### 6. 调度与保留分别摘要

策略列表和表单确认区分别显示“何时备份”和“保留多久”，不将 Cron、执行频率、保留数量混成一行技术信息。原始 Cron 仅在高级模式或辅助信息中出现。

## Risks / Trade-offs

- [Risk] Agent 分布在不同时区时，同一策略表达式会在不同绝对时间执行。 -> 明确标注节点本地时间；独立策略时区留作后续能力。
- [Risk] Cron 识别器错误地把复杂表达式简化后再写回。 -> 仅接受规范化后完全等价的模式，其他表达式始终保持自定义模式和原始值。
- [Risk] 前后端 Cron 解析规则不一致。 -> Master 使用与 Agent 相同的 robfig/cron 配置，并用共享用例覆盖五段、六段、描述符和非法输入。
- [Risk] 用户误以为四个保留数量会相加。 -> 在表单和摘要中说明规则取并集，并给出每日/每周代表性快照示例。
- [Risk] 全零保留规则可能是部分用户的有意配置。 -> 本次选择拒绝全零以防备份立即失去保护；需要无限保留时应另行设计显式选项。

## Migration Plan

1. 先增加前后端校验与 Cron 工具测试，不改变存储结构。
2. 上线可视化构建器和重构后的保留区域；已有策略在打开时进行无损识别。
3. 对无法识别的已有 Cron 默认显示自定义模式，不自动保存或改写。
4. 更新列表摘要和使用文档。

回滚时可恢复原文本框 UI；数据库中的 `schedule` 和 `retention` 数据无需转换。若服务端校验发现历史表达式超出现有 Agent 语义，读取不受影响，仅在再次保存时要求修正。

## Open Questions

- 后续是否需要在 Agent 心跳中上报 IANA 时区，并允许策略覆盖节点时区？
- 是否需要增加“永不自动清理”的显式保留模式，而不是允许四项全为零？
