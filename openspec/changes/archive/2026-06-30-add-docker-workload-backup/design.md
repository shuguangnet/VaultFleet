## Context

VaultFleet 当前的备份策略以 `backup_dirs` 为核心，Agent 侧执行器只理解目录、排除规则、备份模式和归档格式。现有模型适合“主机文件系统备份”，但没有把 Docker 工作负载场景显式建模，因此用户只能手工拼装挂载目录和一致性步骤。对于数据库容器、Compose 应用和 bind mount 目录，这种隐式做法容易造成范围遗漏或误以为系统支持“整容器恢复”。

本次设计要在不改变 VaultFleet 产品定位的前提下，补齐 Docker 场景支持。系统仍然备份主机可见文件与导出产物，不接入 Docker 镜像层备份，也不负责根据备份自动重建容器。技术改动将跨越数据库模型、API、协议、Agent 执行链路与前端策略页，属于典型的跨模块变更。

## Goals / Non-Goals

**Goals:**

- 为策略增加可选 pre/post backup hooks，用于 Docker 工作负载一致性处理。
- 让 Docker 场景的推荐备份范围在 UI 与文档中明确可见。
- 在 Agent 侧将钩子纳入任务执行与失败语义，避免“备份成功但导出失败”的假阳性。
- 保持现有 `snapshot` / `archive` 模式不变，让 Docker 场景成为其上的增强能力，而不是第三种备份模式。

**Non-Goals:**

- 不支持 `docker commit`、`docker save`、镜像仓库导出或容器层快照。
- 不支持自动发现所有容器并自动生成策略。
- 不支持依据备份自动重建容器、网络、端口映射或 Compose stack。
- 不引入对 Docker Engine API 的强依赖，第一版仅执行受控主机命令。

## Decisions

### 1. 将 Docker 支持建模为“现有策略增强”，而不是新备份模式

选择在 `BackupPolicy` / `PolicyPushPayload` 中新增可选钩子字段，例如 `pre_backup_hook` 与 `post_backup_hook`，而不是引入 `docker` 备份模式。理由是当前执行链路已经围绕目录备份组织，Docker 工作负载的核心仍然是“备份挂载数据和编排文件”，只是多了一致性准备步骤。

备选方案是新增 `docker` 模式并在 Agent 中分叉执行器。这个方案概念上更显式，但会迫使 UI、API、协议和任务结果分支扩大，同时掩盖一个事实：实际被保护的仍然是主机文件和导出产物，而不是容器对象本身。

### 2. 钩子使用受控命令定义，并作为任务生命周期的一部分

每个钩子至少包含命令内容与可选超时。Agent 在备份任务上下文内执行命令，继承取消与总超时控制。pre-hook 失败直接终止备份；post-hook 失败将任务标记失败，并在 `TaskResult` / `TaskHistory` 中记录失败来源。

备选方案是把钩子设计成“尽力执行，不影响主任务状态”。这对运维观察不友好，会把导出失败伪装成成功备份，尤其不适合数据库导出场景。

### 3. 第一版仅支持主机级命令，不直接接入 Docker Engine API

实现上通过 `sh -c` 或等效 shell 入口执行受控命令，让用户可以写 `docker exec ...`, `docker compose stop ...`, `mysqldump ...` 这类主机命令。这样能覆盖绝大多数场景，同时避免额外 Docker SDK 依赖、socket 权限处理与 API 兼容性问题。

备选方案是直接调用 Docker API 进行暂停、导出和检查。它能做更强校验，但会把 Agent 权限模型、平台兼容性和测试复杂度整体拉高，不适合当前仓库的第一步落点。

### 4. Docker 场景通过 UI 文案和示例强化，而非自动路径发现

前端策略页增加 Docker 场景说明、示例路径和风险提示，重点告诉用户应备份 volume/bind mount 目录、Compose 文件和 `.env` 等配置。第一版不做自动扫描容器、volume 或 compose 项目。

备选方案是接入 Docker 信息探测自动填充路径。它能改善体验，但会引入在线探测、权限、误报和跨发行版差异，容易让用户误解为“系统已完全理解并可恢复容器拓扑”。

## Risks / Trade-offs

- [钩子具备远程执行能力] → 通过策略校验、显式 UI 文案、超时限制、任务日志脱敏和仅管理员可配置来降低误用风险。
- [post-hook 失败会让已完成的数据备份显示失败] → 在任务结果中区分失败阶段，保留已上传数据事实，同时坚持一致性优先的状态语义。
- [用户误以为支持完整容器恢复] → 在策略页、README 和 capability spec 中重复声明不支持镜像层备份与自动重建。
- [shell 命令跨发行版差异较大] → 第一版只承诺“执行主机命令”，示例以常见 Docker/Compose 命令为主，不做平台无差异保证。

## Migration Plan

1. 为策略模型、API 请求和协议载荷增加可选 hook 字段，保持默认空值兼容旧策略。
2. 前端在不要求旧字段存在的前提下渲染 Docker 场景帮助与 hook 表单。
3. Agent 在收到缺省 hook 字段时保持现有行为不变。
4. 发布说明中强调新能力为可选增强，无需迁移现有策略；只有需要 Docker 一致性控制的节点才需要更新配置。
5. 如需回滚，可停用 UI 入口并忽略 hook 字段；旧策略格式仍然可被当前代码路径处理。

## Open Questions

- hook 字段是否需要结构化为 `{ command, timeout_seconds }`，还是第一版只接受字符串命令并复用策略总超时？
- 是否需要把 hook 执行日志拆分成独立字段，避免与备份 stderr 混杂？
- post-hook 是否应在主备份失败时跳过，还是允许执行清理动作？第一版更倾向仅在主备份成功后执行。
